## 3. SSR payload — `/meditate` is too heavy

The page renders **267 meditation cards** server-side, sending 1.7 MB of HTML on first paint.

### Why it's heavy
- Each card has 10-image srcSet (256w → 3840w) × ~10-15 KB of inline data
- 267 cards × 16.6 KB per card = 4.4 MB raw → compressed to 1.7 MB

### Fix options (any of these, in order of preference)
1. **Paginate**: show 24 per page with "Load more" — 6 KB × 24 = 144 KB (12× smaller)
2. **Virtualize**: use `react-window` or `react-virtuoso` to render only visible cards
3. **Lazy-hydrate cards**: use `next/dynamic` + IntersectionObserver so off-screen cards don't ship HTML
4. **Cut srcSet widths**: remove 1920w and 3840w (CDN already serves a 1200w max)

### Recommendation
**Option 1** (paginate to 24) is fastest to ship — change the page.tsx to use PB `perPage=24` and add a Load More button. A meditation library doesn't need 267 cards above the fold.

## 4. Image srcSet bloat

Every Next.js `<Image>` generates 10 width variants:
```
256w 384w 640w 750w 828w 1080w 1200w 1920w 2048w 3840w
```

The 1920w and 2048w variants are **never used** on the meditation cards (which render at max ~400 px wide on desktop). 

**Fix:** add `sizes="(max-width: 768px) 100vw, (max-width: 1024px) 50vw, 33vw"` (already there) AND configure `next.config.mjs` `images.deviceSizes` to drop 1920/2048/3840. Or, simpler: use the `<picture>` with explicit `<source>` instead of Next/Image for these cards.

## 8. Other small findings

### `/prayer` 404
A real visitor typing `heal.positiveness.club/prayer` lands on a beautifully designed 404 page — but the nav says "Prayers" anyway. Still, search engines and external links may use `/prayer`.

**Fix:** add a redirect from `/prayer` → `/prayers` in `next.config.mjs`:
```js
async redirects() {
  return [
    { source: '/prayer', destination: '/prayers', permanent: true },
    { source: '/library', destination: '/meditate', permanent: true },
    { source: '/library/:path*', destination: '/meditate', permanent: true },
  ];
}
```

### PB `duration_seconds` mostly 0
Many praise records show `duration_seconds: 0` — duration metadata wasn't carried over. This affects any UI that needs to display song length.

**Fix:** run a one-time backfill using `ffprobe` on the CDN files:
```bash
for f in orig-*.mp3; do
  dur=$(ffprobe -v error -show_entries format=duration -of csv=p=0 "$f")
  pb patch $f duration_seconds=$dur
done
```

### Missing `/media/images/site/og-default.png`
The OG image is dynamically generated (`opengraph-image.tsx`) so this isn't a real issue — but if a static fallback is desired, generate a 1200×630 PNG and upload to `heal/images/site/og-default.png` so external link previews have a static asset.
